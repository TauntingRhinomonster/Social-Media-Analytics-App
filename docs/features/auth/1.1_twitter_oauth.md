## 1. Problem Statement

Currently, the engine cannot interact with external social media platforms because we lack a secure mechanism to authenticate user accounts and manage access tokens. The LLM and backend agents are entirely cut off from real-time social data, preventing any analytics or automated actions. Fixing this establishes our foundational data bridge, allowing the engine to verify identities and securely invoke the Twitter/X API on behalf of users.

## 2. Goals

* Successfully negotiate the OAuth 2.0 with PKCE flow with Twitter/X.
* Securely encrypt and persist user access and refresh tokens in our database.
* Provide internal endpoints to retrieve valid tokens for downstream worker services.
* Establish an automated refresh mechanism for expired access tokens.

## 3. Non-Goals

* Fetching historical user timelines or mentions (this belongs in Feature 1.2).
* Implementing authentication for platforms other than Twitter/X (e.g., LinkedIn, Reddit).
* Exposing the token directly to the LLM context.

## 4. Personas & User Stories

* **As a System Integrator**, I want to provide a Twitter authorization link to my users so that they can grant our backend access to their data.
* **As a Backend Data Worker**, I want to retrieve a valid, decrypted access token for a specific user ID so that I can fetch their analytics.
* **As a Security Admin**, I want to ensure tokens are encrypted at rest so that a database breach does not compromise our users' social accounts.

## 5. Functional Requirements

* **FR-1.** The system MUST generate a valid OAuth 2.0 authorization URL with PKCE (Proof Key for Code Exchange).
* **FR-2.** The system MUST intercept the callback URL, exchange the authorization code for an access/refresh token pair, and store it.
* **FR-3.** The system MUST encrypt the access and refresh tokens using AES-GCM before writing to the database.
* **FR-4.** The system MUST automatically attempt to use the refresh token to get a new access token if a Twitter API call returns a 401 Unauthorized status.
* **FR-5.** The system SHOULD emit an observability event when a user successfully connects their account.

## 6. Non-Functional Requirements

* **Performance** — p95 latency targets for token retrieval and decryption must be under 50ms to prevent bottlenecking data workers.
* **Security** — AES-256-GCM encryption is required for all tokens. Keys must be managed via a secure KMS, never hardcoded.
* **Privacy & Compliance** — Adhere to Twitter Developer Agreement regarding token storage and user deletion requests.
* **Accessibility** — Not applicable (backend-only feature).
* **Scalability** — The token retrieval service must handle expected load from highly concurrent background polling workers.
* **Reliability** — 99.9% availability target for token retrieval. Failures in external OAuth must be logged clearly.
* **Observability** — Required metrics: OAuth success rate, OAuth failure rate (by reason), token refresh success/failure.
* **Maintainability** — All cryptography logic must be isolated in a dedicated `pkg/crypto` module.
* **Internationalization** — Not applicable.
* **Backward compatibility** — N/A (Greenfield implementation).

## 7. Acceptance Criteria

* **AC-1.** *Given* an unauthenticated user *When* they request a connection URL *Then* the API returns a valid Twitter OAuth 2.0 URL containing standard scopes (`tweet.read`, `users.read`, `offline.access`).
* **AC-2.** *Given* a valid authorization code from Twitter *When* sent to the callback endpoint *Then* the tokens are successfully exchanged, encrypted, and saved to the database.
* **AC-3.** *Given* an expired access token in the database *When* a worker requests a token *Then* the system securely negotiates a refresh, stores the new tokens, and returns the fresh access token.

## 8. Data Model

* New table: `social_identities`
* Columns: `id` (UUID), `user_id` (UUID, FK), `provider` (Enum: TWITTER), `provider_account_id` (String), `encrypted_access_token` (Text), `encrypted_refresh_token` (Text), `token_expires_at` (Timestamptz), `created_at`, `updated_at`.
* Indexes & constraints: Unique constraint on `(user_id, provider)`. Index on `provider_account_id`.
* Migration file naming convention used by the repo: `server/migrations/001_create_social_identities.sql`.
* Backfill strategy for existing rows: N/A.

## 9. API Surface

* `GET /api/v1/auth/twitter/connect` (Auth: required) — Returns the OAuth URL.
* `GET /api/v1/auth/twitter/callback` (Auth: optional, verified via state parameter) — Handles the redirect code exchange.
* Request & response shapes: standard JSON responses with `{ "url": "..." }` and `{ "success": true }`.
* OpenAPI documentation requirement: MUST be tagged under "Integrations".

## 10. UI / UX

* **Note**: As this is a backend-only toolset, there is no GUI.
* Key user flows (1): Client application hits `/connect`, redirects user to Twitter, user approves, Twitter redirects back to client, client passes code to backend `/callback`.
* Empty / loading / error / offline states: API returns 400 Bad Request with standardized error codes for failed exchanges.

## 13. Dependencies & Sequencing

* Shared infra needed: KMS (Key Management Service) for symmetric encryption keys.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- |
| KMS failure prevents token decryption | L | H | Implement robust retries and caching for the KMS key in memory. |
| Twitter API rate limits on token refresh | M | M | Implement jitter and exponential backoff on refresh attempts. |
| Malformed state parameter in callback | M | L | Strict validation and short TTL for state cache. |

## 15. Rollout Plan

* Feature flag name & default state: `ENABLE_TWITTER_OAUTH` (default: false).
* Migration sequencing: Schema → KMS config → Code → Flip flag.
* Dogfood / pilot cohort: Internal test accounts only for V1.
* GA criteria & comms: Successful end-to-end OAuth flow logged in production without token leakage.
* Rollback path: Turn off feature flag, drop `social_identities` table if necessary.

## 16. Test Plan

* **Unit** — Mocking the Twitter OAuth exchange to ensure proper encryption/decryption logic.
* **Integration** — DB writes/reads for encrypted tokens.
* **End-to-end** — N/A (requires manual browser interaction with Twitter for true E2E).
* **Security** — Ensure database dumps show unreadable gibberish for token columns. Ensure logs redact token payloads.
* **Performance / load** — Benchmark encryption/decryption overhead to ensure it meets the <50ms p95 requirement.
* **Manual exploratory** — Manually authenticate, revoke app access on Twitter, and ensure our backend handles the subsequent rejection cleanly.
